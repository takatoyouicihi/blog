项目背景

建立数据中台背景在这里已经说过了，就不再反复阐述了。

Again，为什么要建立数据中台

一言以蔽之，为了解决效率问题，同时降低创新成本。即：

  • 制定标准/规范、提供高可用数据/算法/应用服务、提供统一、标准的数据研发工具；
  • 减少沟通成本，提升协作效率；
  • 数据资产的集中管控，分布式执行。

如何建立数据中台？

在我看来，数据中台一般会分为三层：

  1. 底层——基础数据层
  2. 中层——业务抽象模型层
  3. 上层——模型算法层

  2. 全域数据清洗

项目情况

接下来按上面7点记录一下搭建的过程。

1.数据仓库的搭建

由于数仓团队也是刚刚开始搭建，因此我也主动去了解目前数仓团队的技术选型，目前采取的是阿里云的Flink+MaxCompute+DataHub。阿里云的体系也方便我和风控团队通过Dataworks进行日常的取数工作，虽然Dataworks的ODPS有点坑就是了…

2.全域数据清洗

这个是真的漫长+痛苦…由于数仓团队的同事不是很了解业务，所以只能由我一个人去制订新的表结构、整理旧数据、梳理切换表时的业务逻辑等等等等。所幸只有API贷款这部分业务的数据不太规范，外加其他业务逻辑比较简单，整个过程大概持续了1个半月。

3.基础数据服务

同上，在清洗的过程中顺带完成了大部分基础数据通用接口的设计，这个只要吃透业务就不难。

4.模型算法服务

风控模型算法在使用时通常需要几百个字段，需要短响应时间以及高吞吐量。数仓团队给出的建议是从ODS层将可能会使用的字段洗成一张大宽表（DWD层）。由于没有太多时间进行调研而且目前模型不算特别多，等有空了再进行调研，看看有没有更好的解决方案。

5.决策指标服务

这个就是典型的反欺诈系统，当用户有欺诈风险时将其拦截，主要应对黑产、撸贷、撸羊毛的团体；由于涉及保密，就此打住。

6.画像标签服务及其上层系统

画像标签服务主要由标签库以及生成用户标签的画像算法组成，基于画像标签服务主要的应用场景目前有ABtest系统。

6.1 标签库

用户画像标签的体系，可以用一棵树状图来表示。该体系通常划分为多个大类（一级分类），每个大类下有多个小分类（二级分类），小分类下面还可以有深度不同的三级、四级等分类。

我将大分类分为以下几类：

  • 账号属性特征。即用户个人的固有属性，如性别年龄等；
  • 第三方数据特征。即用户个人在第三方数据源中的表现，如芝麻分、同盾多头等；
  • 账号行为特征。即用户的行为轨迹；
  • POI特征。即用户对不同资方的表现；
  • LBS特征。即用户实时地理位置。

6.2画像生成

画像生成分为五部分：

  1. 数据集成

    这部分主要依赖数仓团队把埋点数据、第三方数据源数据、业务数据等集成汇集到数据仓库里。

  2. 特征计算

    这部分主要的工作是数据处理和数据加工。

  3. 标签库维护

    针对标签的收录管理，我们开发了管理后台来负责各种标签基于层级划分的收录，以及标签的数量和质量监控；而对于数据的使用，总的来说主要有两种模式：给定设备ID查询用户属性标签，给定用户属性的组合（交集、并集、取反等），圈定出符合条件的一批用户。

  4. 算法模型

  5. 画像接口

    为上层服务提供统一的画像接口。

6.3ABtest系统

基于画像标签服务，我们立即投入AB测试系统的设计开发。我认为一个ABtest系统应该具备以下4个条件：

  1. 优化场景
  2. 量化指标
  3. 用户稳定
  4. 长期反馈
